# Tier0 Marimo 部署方案 (AK/SK 凭证)

## 一、方案概述

### 1.1 业务需求

Tier0 项目需要在 Kubernetes (AWS EKS) 环境中部署 Marimo Notebook 服务，核心要求：

| 需求               | 说明                                                     |
| ------------------ | -------------------------------------------------------- |
| **S3 Tables 连接** | Notebook 能够直接查询 AWS S3 Tables 中的 Iceberg 表      |
| **多租户隔离**     | 每个租户只能看到自己 namespace 下的表，实现数据隔离      |
| **AWS 凭证管理**   | 使用 AK/SK (Access Key/Secret Key) 静态凭证，配置简单稳定 |
| **稳定可维护**     | 便于版本管理、问题排查和后续升级                         |

### 1.2 技术挑战

| 挑战                    | 详细说明                                                                     |
| ----------------------- | ---------------------------------------------------------------------------- |
| **数字 Namespace 问题** | S3 Tables 的 namespace 是数字（如 `326933050398736`），在 SQL 中需要引号包裹 |
| **表列表过滤**          | Marimo 侧边栏默认显示所有表，需要按租户 namespace 过滤                       |
| **DuckDB 连接共享**     | `sitecustomize.py` 初始化的 S3 Tables 连接需要被 `get_datasets.py` 使用      |
| **凭证管理**            | 使用长期有效的 AK/SK 凭证，无需处理凭证过期问题                              |

### 1.3 方案优势

相比 IRSA 凭证刷新方案，AK/SK 方案具有以下优势：

| 优势 | 说明 |
|------|------|
| **配置简单** | 只需设置环境变量，无需配置 ServiceAccount、IAM Role 等 |
| **稳定可靠** | 凭证长期有效，无需后台刷新线程，避免内存泄漏 |
| **易于调试** | 凭证问题一目了然，不涉及复杂的 credential chain |
| **资源占用低** | 无后台线程，内存占用更低 |

### 2.2 核心组件

#### 2.2.1 get_datasets.py (Patch)

**位置**: `marimo/_data/get_datasets.py`

**修改内容**:

```python
# 1. 添加引号处理函数（处理数字 namespace）
def _quote_identifier(name: str) -> str:
    """Quote identifier if it starts with a digit or contains special characters."""
    if re.match(r"^[a-zA-Z_][a-zA-Z0-9_]*$", name):
        return name
    return f'"{name}"'

# 2. 读取环境变量
_S3TABLES_NAMESPACE = os.getenv("NAMESPACE_ID")
_S3TABLES_DATABASE = os.getenv("S3TABLES_DATABASE", "s3tables")

# 3. 过滤其他 namespace 的表
if _S3TABLES_NAMESPACE and database == _S3TABLES_DATABASE:
    if schema != _S3TABLES_NAMESPACE:
        continue  # 跳过其他租户的表

# 4. 使用 default_connection 确保连接共享
def execute_duckdb_query(connection, query):
    return _get_default_connection().execute(query).fetchall()

# 5. 处理 Iceberg catalog 表的数字 namespace
qualified_name = f"{database}.{_quote_identifier(schema)}.{_quote_identifier(name)}"
```

#### 2.2.2 sitecustomize.py

**位置**: `tier0/sitecustomize.py` → 安装到 `site-packages/`

**作用**: Python 启动时自动执行，初始化 S3 Tables 连接

**凭证策略**:
- 使用 **AK/SK 静态凭证**（从环境变量读取 `AWS_ACCESS_KEY_ID` 和 `AWS_SECRET_ACCESS_KEY`）
- 凭证长期有效，无需刷新机制
- 配置简单，适合生产环境

```python
def _early_init_s3tables():
    """在 Python 启动时初始化 S3 Tables 连接"""

    # 1. 读取环境变量
    bucket_arn = os.getenv("S3TABLES_BUCKET_ARN")
    aws_access_key = os.getenv("AWS_ACCESS_KEY_ID")
    aws_secret_key = os.getenv("AWS_SECRET_ACCESS_KEY")
    namespace_id = os.getenv("NAMESPACE_ID")

    # 2. 使用 DuckDB 默认连接
    conn = duckdb.default_connection()
    conn.sql("INSTALL iceberg; INSTALL aws; LOAD iceberg; LOAD aws;")

    # 3. 配置 AWS 凭证（使用 AK/SK）
    conn.sql(f"""
        CREATE SECRET aws_s3tables (
            TYPE S3,
            KEY_ID '{aws_access_key}',
            SECRET '{aws_secret_key}',
            REGION 'ap-southeast-1'
        );
    """)

    # 4. 附加 S3 Tables
    conn.sql(f"ATTACH '{bucket_arn}' AS s3tables (TYPE ICEBERG, ENDPOINT_TYPE s3_tables);")

    # 5. 设置默认 schema
    conn.sql(f'USE s3tables."{namespace_id}";')

    # 6. 存储到 builtins 供其他模块使用
    builtins._tier0_s3conn = conn
    builtins._tier0_credential_method = "ak_sk"
```

#### 2.2.3 tier0_s3tables.py

**位置**: `tier0/tier0_s3tables.py` → 安装到 `site-packages/`

**作用**: 提供 `s3conn` 对象供 Notebook 用户使用

**特性**:
- 自动使用 `sitecustomize.py` 初始化的连接
- 使用 AK/SK 静态凭证，无需凭证刷新
- 如果 `sitecustomize.py` 未初始化，会自动初始化连接

```python
def _create_connection() -> duckdb.DuckDBPyConnection:
    """获取或创建 S3 Tables 连接"""

    # 优先使用 sitecustomize 创建的连接
    if hasattr(builtins, "_tier0_s3conn"):
        return builtins._tier0_s3conn

    # 如果未初始化，自行初始化
    conn = duckdb.default_connection()
    _attach_s3tables(conn)  # 使用 AK/SK 配置并 ATTACH
    return conn

# 导出连接
s3conn = _create_connection()  # DuckDB 连接（Marimo 侧边栏识别）
```

**用户在 Notebook 中使用**:

```python
from tier0_s3tables import s3conn

# 查询数据
df = s3conn.sql("SELECT * FROM my_table LIMIT 10").df()
```

## 三、环境配置

### 3.1 环境变量

| 变量名                    | 必填 | 说明                    | 示例                                                   |
| ------------------------- | ---- | ----------------------- | ------------------------------------------------------ |
| `S3TABLES_BUCKET_ARN`     | ✅   | S3 Tables 桶 ARN        | `arn:aws:s3tables:ap-southeast-1:123:bucket/my-bucket` |
| `NAMESPACE_ID`            | ✅   | 当前租户的 namespace ID | `326933050398736`                                      |
| `AWS_ACCESS_KEY_ID`       | ✅   | AWS Access Key ID       | `AKIAIOSFODNN7EXAMPLE`                                 |
| `AWS_SECRET_ACCESS_KEY`   | ✅   | AWS Secret Access Key   | `wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY`             |
| `S3TABLES_DATABASE`       | ❌   | DuckDB 中的数据库名     | `s3tables`（默认值）                                   |
| `AWS_REGION`              | ❌   | AWS 区域                | `ap-southeast-1`（默认值）                             |

### 3.2 Kubernetes Deployment 示例

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: notebook
  namespace: tenant-326933050398736
spec:
  replicas: 1
  selector:
    matchLabels:
      app: notebook
  template:
    metadata:
      labels:
        app: notebook
    spec:
      containers:
      - name: notebook
        image: 350646758444.dkr.ecr.ap-southeast-1.amazonaws.com/tier0/tier0-marimo:latest
        ports:
        - containerPort: 2718
        env:
        - name: S3TABLES_BUCKET_ARN
          value: "arn:aws:s3tables:ap-southeast-1:350646758444:bucket/abc123/tier0-s3tables"
        - name: NAMESPACE_ID
          value: "326933050398736"
        - name: AWS_ACCESS_KEY_ID
          valueFrom:
            secretKeyRef:
              name: aws-credentials
              key: access-key-id
        - name: AWS_SECRET_ACCESS_KEY
          valueFrom:
            secretKeyRef:
              name: aws-credentials
              key: secret-access-key
        - name: AWS_REGION
          value: "ap-southeast-1"
        resources:
          limits:
            memory: "2Gi"
            cpu: "2"
          requests:
            memory: "1Gi"
            cpu: "500m"
```

## 四、构建与部署

### 4.1 EC2 构建环境

**推荐配置**:
- **位置**: EC2 (ap-southeast-1，与 ECR 同区域)
- **实例**: c6i.2xlarge (8 vCPU, 16GB RAM)
- **存储**: 100GB gp3 SSD
- **优势**: 内网推送到 ECR，速度提升 10-50 倍

**EC2 初始化**:
```bash
# 安装 Docker
sudo yum update -y
sudo yum install -y docker git
sudo systemctl start docker
sudo systemctl enable docker
sudo usermod -aG docker ec2-user

# 克隆代码
git clone https://github.com/FREEZONEX/marimo.git
cd marimo
git checkout tier0-patch
```

### 4.2 构建镜像

```bash
cd /root/cyw/coding/marimo

# 登录 ECR
aws ecr get-login-password --region ap-southeast-1 | \
  docker login --username AWS --password-stdin \
  350646758444.dkr.ecr.ap-southeast-1.amazonaws.com

docker build -f docker/Dockerfile \
  --target tier0 \
  -t 350646758444.dkr.ecr.ap-southeast-1.amazonaws.com/tier0/tier0-marimo:latest \
```

## 八、常见问题

### Q1: 如何更换 AWS 凭证？

```bash
# 更新 Kubernetes Secret
kubectl create secret generic aws-credentials \
  --from-literal=access-key-id=NEW_ACCESS_KEY \
  --from-literal=secret-access-key=NEW_SECRET_KEY \
  -n tenant-326933050398736 \
  --dry-run=client -o yaml | kubectl apply -f -

# 重启 Pod 使新凭证生效
kubectl rollout restart deployment notebook -n tenant-326933050398736
```

### Q2: 如何切换到不同的 S3 Tables Bucket？

修改 Deployment 中的 `S3TABLES_BUCKET_ARN` 环境变量，然后重启 Pod。

### Q3: 如何调试 S3 Tables 连接问题？

```bash
# 1. 检查环境变量是否正确
kubectl exec <pod-name> -n <namespace> -- env | grep S3TABLES

# 2. 检查 AWS 凭证是否有效
kubectl exec <pod-name> -n <namespace> -- \
  python -c "import boto3; print(boto3.client('sts').get_caller_identity())"

# 3. 查看详细的初始化日志
kubectl logs <pod-name> -n <namespace>
```

### Q4: 相比 IRSA 方案有什么区别？

| 方面 | IRSA 方案 | AK/SK 方案 (当前) |
|------|-----------|-------------------|
| **凭证类型** | 临时凭证 (12小时有效期) | 长期静态凭证 |
| **配置复杂度** | 高（需要 ServiceAccount、IAM Role） | 低（只需环境变量） |
| **凭证刷新** | 需要后台线程刷新 | 无需刷新 |
| **内存占用** | 较高（有后台线程） | 较低 |
| **稳定性** | 可能因刷新失败导致问题 | 更稳定 |
